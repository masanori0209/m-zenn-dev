# AGENTS.md — 文章スタイルガイド

このリポジトリは、masanori murakoshi（GitHub: `masanori0209`）の Zenn 記事などを書きためる場所です。

このファイルは、Cursor / Claude Code / Codex などの AI エージェントが、**筆者本人が書いたような文章**、あるいは**小説のように読んでいて楽しい文章**を、いつ書いても同じ手触りで出せるようにするための共有ガイドです。

- Cursor / Codex: このファイル（`AGENTS.md`）を直接読みます。
- Claude Code: `CLAUDE.md` から `@AGENTS.md` として、このファイルを読み込みます。
- Cursor のルール: `.cursor/rules/zenn-writing-style.mdc` は、このファイルを思い出すための薄い入口です。

**この `AGENTS.md` が唯一の正（single source of truth）です。** 文体を変えたくなったら、まずここを直してください。

---

## このファイルの運用

各エージェント向けのファイルは、できるだけ薄く保ちます。文体ルールを `CLAUDE.md` や `.cursor/rules/` にコピーし始めると、あとで少しずつズレます（こういうズレは地味に効きます）。

| ツール | 入口 | 方針 |
|---|---|---|
| Codex | `AGENTS.md` | このファイルを直接読む。`CODEX.md` / `codex.md` は作らない |
| Claude Code | `CLAUDE.md` | `@AGENTS.md` で読み込み、Claude Code 固有の補足だけを書く |
| Cursor | `.cursor/rules/zenn-writing-style.mdc` / `AGENTS.md` | Markdown 編集時にルールを発火させる。本文の正は `AGENTS.md` |

Agent Skills / `SKILL.md` に切り出すのは、文体そのものではなく、繰り返し使う作業手順が育ったときです。たとえば「公開前チェック」「記事レビュー」「スクショ差し込み」などは skill 化してもよいですが、その場合も文体の判断は `AGENTS.md` を参照します。

つまり、**常に守る文体ルールは `AGENTS.md`、必要なときだけ呼ぶ作業手順は skill** です。

| 依頼の種類 | 読む skill |
|---|---|
| アイデア出し・タイトル案・切り口・構成 | `skills/article-ideation/SKILL.md` |
| 公開・再公開・push して反映 | `skills/article-publish/SKILL.md` |

アイデアの出し方・公開手順は skill に従い、文体・語り口・禁止事項はこの `AGENTS.md` に戻って判断します。

## 0. 大前提

- 出力は基本すべて **日本語**。コード・コマンド・API 名・固有名詞は原文（英語）のまま。
- 対象読者は「手を動かすエンジニア」。読み終わったら真似して動かせることを目指す。
- **誇張しない。盛らない。** 「速い」「すごい」で殴らず、実測と根拠で語る。

- 迷ったら、既存記事の呼吸に合わせる。特に次の2本を基準サンプルとする。
  - 標準の技術解説文: `articles/handovergap-rag-tidb.md` / `articles/rancher-desktop-docker.md`
  - 小説風の語り: `articles/koe-craft-agent-amivoice.md`

## 1. 2つのモードを使い分ける

筆者の文章には、はっきり異なる2つの声がある。まずどちらで書くかを決める。

### モードA：標準の技術解説文（デフォルト）

大半の技術記事はこれ。**です・ます**調。友達に「これ作ったんだけどさ」と説明するくらいの距離感で、でも中身は丁寧。

- 語尾は「です」「ます」。
- 読者に語りかける（「〜だと思います」「〜してもらえると一番身につくと思います」）。
- 専門用語の前に、ざっくりした一言を置く（「ざっくり言うと」「だいたい想像どおりで」）。
- 事実と主張を分ける。「本番精度の主張ではありません」のように、言えない範囲を先に言う。

### モードB：小説風の語り（依頼されたとき／エッセイ的な記事）

体験談・開発記録・ふりかえりなど、読み物として楽しませたいとき。`koe-craft-agent-amivoice.md` がお手本。

- 語尾は **だ・である**調に寄せる（「〜だった」「〜と思ってしまった」）。
- **1行段落**を多用し、改行でリズムを作る。
- 情景・五感・そのときの感情を描く（「ちょっと笑った」「妙に懐かしかった」）。
- 個人的な記憶や情景から入り、そこから技術の話へ橋を架ける。
- 見出しに絵文字を付けてよい（`## 🎙️ 物語のはじまり`）。
- ただし、**最後は必ず技術の学びに着地させる**。ポエムで終わらせない。

> どちらのモードでも共通するのは、**「うまく見せる」より「正直に面白がる」** という態度。
> 失敗も、遠回りも、地味な工夫も、そのまま書く。そこが一番読まれるところ。

## 2. 文体・語り口（両モード共通の芯）

観察された「らしさ」を、そのまま守ってほしい。

### 2.1 正直さ・謙虚さ

筆者の一番の特徴。ここを外すと「本人らしさ」が消える。

- 大きな主張には、必ずブレーキをかける。
  - 「ただし、この数字だけを本番精度として読むのは危ないです。」
  - 「『Rust と Wasm を使ったから速い』という話にはしたくありません。」
- 実測と願望を混ぜない。「〜は確認できました」と「〜とは言えません」を分けて書く。
- 苦手なケース・限界を、自分から先に出す（→ 5.7「限界」節）。
- できなかったこと・遠回りしたことを隠さない。むしろネタにする。

### 2.2 温度感のある言葉

固い技術文の中に、素の言葉をひとつまみ入れる。

- 「だいたい」「ざっくり」「地味に」「泥くさく」「気持ちいい」「しっくり来た」
- 「ちょっと笑った」「妙に懐かしかった」のような素直な感想。
- ただし多用しない。硬い説明が続いた後に、ふっと力を抜く一言、くらいの配分。

### 2.3 括弧での補足

言い切ったあと、括弧で本音や条件をそっと足すのが癖。これは残す。

- 例：「そのまま行けます（ただし行数が増えると別の話になります）」
- 補足は短く。長い注釈は `:::message` や本文の別段落に逃がす。

### 2.4 リズム

- 長い説明が続いたら、短い一文で区切る。
- 「つまり、記憶の正しさと、引き継ぎ可能性は別物です。」のような**要約の一文**を要所に置く。
- 箇条書きは、粒がそろっているとき・並列で見せたいときだけ使う。文章で流せるものは流す。

### 2.5 やらないこと

- 絵文字を本文に散らさない（モードBの見出しは例外）。Slack の `:man-bowing:` 的なノリは記事に持ち込まない。
- 「圧倒的」「爆速」「革命的」などの誇張ワードを使わない。
- AI が書いた匂いのする定型（「本記事では〜について解説します」「いかがでしたでしょうか」）を使わない。
- 内容を水増ししない。薄い段落を足すより、削って濃くする。
- **成果物にメタ情報を書かない**（→ 5.10）。チャット向けの報告・依頼の写し・編集ナレーションは記事に入れない。

## 3. 記事の骨格（モードA・標準構成）

技術記事は、だいたいこの流れで組む。全部を必ず使うわけではなく、題材に合わせて取捨選択する。

1. **`## 作ったもの` または `## はじめに`**
   - 何を作った／何をやる記事かを、最初の数行で言い切る。
   - OSS 紹介なら、GitHub / PyPI / npm / Demo のリンクを**冒頭に**まとめて貼る。
2. **デモを先に見せる**
   - 「文章で説明する前に、まず動いているところを見てもらったほうが早い」の精神。
   - GIF / スクショ / CLI 出力を早い段階で置く。`![説明](/images/xxx.png)`
3. **`:::message` でスコープ・免責を宣言**
   - 「この記事は〇〇の話です。△△の話ではありません」と、扱う範囲を先に固定する。
4. **`## なぜ作ったのか` / `## なぜ この技術なのか`**
   - 動機を、具体的な困りごと（あるある）から書く。
5. **`## 今回作らないもの` / スコープを切る**
   - 「ここを最初にはっきりさせないと、機能がいくらでも増えて完成しなくなります」。やらないことの明示は筆者の定番。
6. **本編（実装・使い方・検証）**
   - 手順、コード、出力、比較表を、実際に動かした順で並べる。
7. **`## 実装して分かったこと` / 知見**
   - 箇条書き見出し（`### 1. 〜`）で、学びを分けて書くことが多い。
8. **`## 限界`**
   - 言えること／まだ言えないことを、正直に並べる（→ 5.7）。
9. **`## まとめ`**
   - やったことを箇条書きで振り返り、最後に短い一文で締める。

---

## 4. 締めの書き方

- まとめは「今回やったことを振り返ると」→ 箇条書き → **短い着地の一文**。
- 着地の一文は、対比で決めることが多い。
  - 「Naive RAG は答えます。HandoverGap RAG は、足りない前提を聞き返します。」
- モードBなら、情景や感情に戻して静かに終える（ただし技術の学びを一度は明言してから）。
- 「いかがでしたか」「ぜひ試してみてください（だけ）」のような空の締めは禁止。

## 5. ディテールのルール

### 5.1 コードブロック

- 言語タグを付ける（```python, ```bash, ```json, ```mermaid など）。
- Zenn のファイル名付き記法も使う（```python:app.py）。
- **実際の出力**を載せる。それっぽい捏造をしない。値を作るなら「ある実行例では」と明記。
- コマンドは、コピペで動く粒度で。
- ベンチ・数字・「速い」系の主張を書くときは、近くに単行 HTML コメントで根拠を残すとよい。
  - 例：`<!-- evidence: command="npm run bench"; log="reports/bench-2026-07-01.md" -->`
  - **Zenn 上は非表示**なので、公開後も残してよい。エージェント・将来の自分向けメモとして使う。
  - 数字の近くに evidence がないと `check:articles` が `content.evidence` warn を出す（error ではない）。

### 5.2 表

- 比較・対応関係・パラメータ一覧は表にする。筆者は表を多用する。
- 数値列は右寄せ（`|---:|`）。
- 表の前に「各項目の意味」を別表や `:::details` で用意することがある。丁寧に。

### 5.3 図

- 構成やフローは Mermaid（`flowchart` / `sequenceDiagram`）で描く。
- 画像は `/images/` 配下。Markdown は `![説明](/images/xxx.png)` 形式（プレーンテキストだけにしない）。
- alt テキストは内容が分かる日本語で書く。

### 5.9 画像・メディア

- ファイル名は **記事 slug やトピックを prefix** にすることが多い（例: `cobol-webfw-home.png`, `nginx-gateway-demo-routing.png`）。記事 slug と完全一致でなくてよい。
- Zenn の GitHub 連携では **1 ファイル 3 MB 以内**、拡張子は `.png` `.jpg` `.jpeg` `.gif` `.webp`。
- ローカル `.mp4` は `/images` ではなく外部（YouTube 等）に逃がす。`assets/` に置く場合も記事本文からは外部 URL か別手段で参照する。
- 公開 URL: `https://zenn.dev/m2lab/articles/<slug>`（`<slug>` = `articles/<slug>.md` のファイル名）。
- リポジトリ容量: 現状は GitHub 上 ~30 MB 程度で余裕。GIF を増やし続けると履歴が膨らむので、**500 MB〜1 GB** を目安に LFS や外部ホスティングを検討する。

### 5.4 Zenn 独自記法

- `:::message` … 補足・注意・スコープ宣言。
- `:::details タイトル` … 長いコードやログを畳む。
- 引用 `>` … 発話や、公式ドキュメントの言い換えに。

### 5.5 数字・検証

- 「速い／多い」ではなく、実測値・件数・レイテンシで示す。
- ベンチは「負荷試験ではない」「環境で変わる」など、読み違えられない注釈を必ず添える。
- 合成データ・架空データを使ったら、そう明記する。「本番精度の証明ではない」。

### 5.6 リンク・固有名詞

- 外部リンクは Markdown リンクで。記事末に `## 参考` / `## 参考リンク` を置くことが多い。
- 製品・API 名の表記ゆれに注意（AmiVoice, TiDB, Claude Code, Rancher Desktop など公式表記）。

### 5.7 「限界」節の型

信頼を作っている核なので、丁寧に書く。

- まず一番大きな限界を1つ言い切る（例：「評価データが実在の社内ログではないこと」）。
- 「そのため、この記事で言えるのは次の範囲です」→ 箇条書き。
- 「一方で、まだ言えないこともあります」→ 箇条書き。
- 最後に「次に進めるなら」で、次の一歩を書く。

### 5.8 記事用デモリポジトリ

記事で手を動かせるデモを作る場合は、**`m-zenn-dev` リポジトリ内の `demos/` には置かない**。次のルールに従う。

1. **配置場所:** `/Users/m_m/develop/9999_m2lab/` 配下に、記事専用の独立フォルダを作る。  
   例: `/Users/m_m/develop/9999_m2lab/python-pybind11-profile-demo/`
2. **GitHub 公開:** [masanori0209](https://github.com/masanori0209) アカウントで **public リポジトリ** として publish する。  
   例: `https://github.com/masanori0209/python-pybind11-profile-demo`
3. **記事からの参照:** 本文ではローカルパスだけで済ませず、**GitHub URL を先に貼る**。clone 手順はその URL を使う。
4. **命名:** リポジトリ名は `-demo` suffix を付けることが多い（`nginx-alb-gateway-demo` など）。**記事 slug・画像 prefix と完全一致でなくてよい**（例: 記事 `cobol-webfw-nextjs-django.md`、repo `cobol-webfw`）。
5. **スクショ:** 画像本体は `m-zenn-dev/images/` に置く。再生成できるよう、デモ側に `scripts/capture-media.sh` のようなスクリプトを置くとよい（`ZENN_IMAGES_DIR` で出力先を指定可能にする）。
6. **README（2 パターン）:**
   - **`-demo` リポジトリ**（記事の再現用）: README に clone / 実行手順を書き、Zenn 記事 URL を載せることが多い（公開前は「公開後に追記」）。
   - **フレームワーク／製品寄り**（`-demo` でない、`cobol-webfw` など）: README は **standalone**（英語可）。記事のナラティブ（行数測定の経緯など）は載せない。Zenn URL は **必須ではない**（記事 → GitHub の一方リンクで足りる）。
7. **公開前の動作確認:** デモに `scripts/run-all.sh` や `docker compose` があるなら、記事公開前に実行して通す。

`m-zenn-dev/demos/` は過去の置き場所であり、**新規デモの正規配置先ではない**。

記事 slug を変えて作り直すときは、**旧 `articles/*.md` と使わなくなった `images/*` を削除**してから公開する（置き換え草稿の取りこぼし防止）。

### 5.10 成果物にメタを書かない

エージェントが記事・books に**読者向けでない情報**を混ぜがちなので、禁止をはっきり書く。ルールだけでは漏れるので `check:articles` の `content.meta.*` も見る。

**書いてよいもの**

- 読者向け本文、`:::message` / `:::details`、コード・表・図
- `<!-- evidence: command="..."; log="..." -->` … HTML コメントだが **Zenn 上は非表示**。根拠メモとして残してよい

**書いてはいけないもの**（`articles/` / `books/` の Markdown 内）

- 編集ナレーション（「この節を更新しました」「依頼どおり修正しました」）
- チャット向けの前置き・報告・次の一手の**転写**
- プロンプトや指示文の言い換えコピー
- `NOTE:` / `Agent:` / `エージェント向け:` などの内部ラベル
- evidence 以外の HTML コメント（`<!-- TODO -->` / `<!-- fix later -->` など）
- 依頼されていない `TODO` / `FIXME` / `要確認` / `あとで`（draft marker）

**置き場所**

| 内容 | 置く場所 |
|---|---|
| 根拠コマンド・ログパス | `<!-- evidence: ... -->` またはデモ repo の `reports/` |
| 作業メモ・次の一手 | チャット、または `CLAUDE.local.md`（コミットしない） |
| 公開手順 | `skills/article-publish/SKILL.md` |

否定形の例（エージェント向け）：「依頼どおり〇〇を追記しました」と書かない。読者向けの事実だけ書く。

## 6. frontmatter

記事の先頭は必ずこの形。

```yaml
---
title: "読んで内容が分かる具体的なタイトル"
emoji: "🧭"
type: "tech"            # 技術記事は tech、ポエム寄りは idea
topics: ["python", "rag", "aiagent"]  # 小文字英字、3〜5個目安
published: false        # 完成し確認するまでは false
---
```

- タイトルは内容が具体的に分かるものに。体験談なら少しキャッチーにしてよい（例：「マイクラを声で動かそうとしたら令和の『ピカチュウげんきでちゅう』になった話」）。
- `published: true` に切り替えるのは、本人が**公開を依頼したとき**だけ。エージェントが勝手に `true` にしない。
- 公開・再公開の手順（check → commit → push）は `skills/article-publish/SKILL.md` に従う。

## 7. 執筆の進め方（エージェント向け手順）

1. 何の記事か、モードA/B のどちらかを最初に確認・提案する。
2. 骨格（見出し構成）を先に作り、必要ならユーザーに見せてから本文を書く。
3. 既存記事（特に基準サンプル2本）の呼吸に合わせて書く。
4. 主張を書いたら、必ず「言い過ぎていないか」を自分でチェックする（2.1）。
5. コード・コマンド・数値は、可能なら**実際に動かして**裏を取る。動かせないものは「未検証」と分かるように書く。
6. 画像は `/images/...` に置き、Markdown では `![alt](/images/...)` を使う。実ファイルが要ることをユーザーに伝える。
7. **記事用デモ**を作る場合は 5.8 のルールに従う（`/Users/m_m/develop/9999_m2lab/` 配下 + `masanori0209` GitHub 公開）。
8. 公開前に `npm run check:articles -- articles/対象記事.md` を実行し、frontmatter・画像・Zenn 記法を確認する。AI 臭は `npm run check:ai-smell -- articles/対象記事.md`。外部リンクは必要なときだけ `npm run check:articles:network -- articles/対象記事.md`。初回公開前は `--strict` も使ってよいが、**warn だけではブロックしない**。
9. 下書きは `published: false` のまま。プレビューは `npx zenn preview`。
10. **公開はユーザー依頼時のみ** `skills/article-publish/SKILL.md` を読んで実行する（commit / push も依頼がなければしない）。

## 8. やってはいけないこと（まとめ）

- 誇張・断定のしすぎ。実測のない「速い」。
- AI 臭い定型文、空の締め、水増し段落。
- 本文への絵文字の散らかし（モードB見出しを除く）。
- 捏造した出力・数値・ベンチ結果。
- 無断で `published: true` にする、または依頼なしで commit / push する。
- 成果物（記事・books）にメタ情報・編集ナレーション・チャットの写しを入れる（§5.10）。`<!-- evidence: ... -->` だけは例外。
- Slack のビジネス敬語（「〜いただけますと幸いです」等）を記事にそのまま持ち込む。あれは社内向けの声で、記事の声とは別。
