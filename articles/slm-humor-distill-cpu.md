## 😏 物語のはじまり

手元の MacBook だけで、モデルって作れないのかな。

ふと、そう思った。

最近、なけなしのお金をかき集めて買った MacBook Air がある。M5。メモリ 32GB。薄いアルミの板越しに、自分の顔がぼやけて映る。クラウドの LLM も、重い GPU サーバも、いまの懐事情だとあまり触りたくない。この板に何か生やしたい。
<!-- evidence: command="system_profiler SPHardwareDataType"; log="reports/hw.txt" -->

買ったとき、妻はどぎつく怒った。

怒った理由は明確だった。お金がないのに、なぜまた薄い板を増やしたのか。いまある薄い板は十分薄くないのか。

私は、薄さは同じでもメモリが違う、と言った。

その反論の回収期限は、まだ来ていない。

だったら、炙るしかない。API に課金して「使いました」と言うより、この筐体そのものでモデルを動かしたほうが、弁明の材料になる気がした。たぶん。多分。いや、甘い。でも、甘いこと自体が MacBook Air の仕様なので、ゆるしてほしい。

「作る」の意味はいろいろある。ゼロから事前学習するのは、もちろん無理だ。地球の終わりまで計算して、家計は先に終わる。

でも、**小さなモデルを育てて、ローカルで笑わせられるか**、なら試せる気がした。

どうせなら、タスクもふざけた。「くすっと笑える一言」を返す相棒。要約でも分類でもない。正解率は測れない。測れないことは、妻への説明にも使える。

Sakana AI の [TinySwallow-1.5B](https://huggingface.co/SakanaAI/TinySwallow-1.5B) を小さく真似るつもりで、先生に 14B、生徒に 1.5B を置いた。

妻への弁明にはならないかもしれない。ただ、なけなしの薄い板が、ただの薄い板ではないことは、せめて証明したかった。

## 育てる前：1.5B は真面目すぎた

まず素の生徒に話しかけてみた。

お題は「電子レンジが壊れた」。

返ってきたのはこれ。

> 電子レンジが壊れたなんて、ちょっとショックかも。でも、諦めずに手当しましょう。例えば、新しい電子レンジを手に入れるかもしれませんよ。

励ます。優しい。でも **くすっとはしない**。

傘を忘れた、と言っても「やばい、傘忘れたよ」程度。小型モデルのご愛嬌というか、地味な正直さというか、そういう感じだった。

これが before。


| お題           | before（素の1.5B）   | after（蒸留後）                 |
| ------------ | ---------------- | -------------------------- |
| 傘を忘れた        | やばい、傘忘れたよ。       | 雨の中を歩くなんてまるでネコが泳ぐようなもんだね   |
| スマホの充電が3%    | 慌てず、すぐに新品に換えよう！  | 電池が減っても心はいつも満充電でいられるもんね    |
| 満員電車で靴紐がほどけた | 二重に穿いた靴が楽に脱けそうだ。 | 靴紐ほどけるなんて 満員電車の隙間探しより難しいねえ |

## 「作る」を分解する


| 意味                          | 現実性（M5 32GB） |
| --------------------------- | ------------ |
| 既存モデルをローカル推論する              | ◎            |
| 小さなモデルをタスク向けに「育てる」（蒸留・LoRA） | ○            |
| 7B 級をゼロから事前学習する             | ×            |


今回やったのは真ん中で、しかも目的は汎用の賢さではなく、**笑いに寄せる**ことだけだ。

## 今回作らないもの

- 7B 級の事前学習
- 汎用チャット性能の底上げ
- 厳密な TAID / logit 蒸留の再現
- 本番品質のユーモア評価

## 🧑‍🏫 先生が、まず面白くなかった

蒸留の流れは単純だ。先生に「お題 → くすっと返し」を大量生成させ、そのお手本で生徒を LoRA 微調整する。

```mermaid
flowchart LR
  T["先生 14B\nお題→返答を生成"] --> D["教師データ\n(JSONL)"]
  D --> L["生徒 1.5B\nMLX LoRA 微調整"]
  L --> F["マージ / 量子化"]
  F --> I["推論\n(M5 ローカル)"]
```



問題は、先生の出来が安定しないことだった。

Qwen は多言語モデルなので、温度を上げると日本語のお題なのに、別の言語に滑る。

```text
傘忘れたって、傘の中身忘れなきゃ大丈嘛。毕竟，里面装的都是干劲儿！
「了解しました、ただ今駆けつらー、밥ункь_SAVE」
```

ちょっと笑った。笑いどころというより、**先生が先に壊れている**感じ。

対策は地味だった。温度を 0.7 に下げ、プロンプトで「日本語だけ」と念押しし、ハングルや英語が混ざった行は機械的に捨てる。

```python
def is_clean_japanese(text: str, max_len: int = 80) -> bool:
    if not text or len(text) > max_len:
        return False
    if re.search(r"[\uac00-\ud7a3]", text):   # ハングル
        return False
    if len(re.findall(r"[A-Za-z]", text)) > 6:
        return False
    if len(re.findall(r"[\u3040-\u30ff]", text)) < 3:
        return False
    return True
```

20 お題 × 6 回サンプリングして、**104 件だけ採用**した。16 件は捨てた。

先生の面白さが天井になる、というのはここで体感した。生徒がそれを超えるわけがない（ただし「寄せる」には効く）。

## 環境

学習も推論も [mlx-lm](https://github.com/ml-explore/mlx-lm)。Apple Silicon の Metal を叩く。CUDA も PyTorch も要らない。

```bash
uv venv --python 3.12 .venv
source .venv/bin/activate
uv pip install -r requirements.txt
```

:::message
ハマりどころ: `mlx-lm 0.31.3` は `transformers 5.x` と非互換。`transformers<5` に固定すると通った。
:::

## 🔧 生徒を LoRA で育てる

`mlx-lm` の LoRA で、assistant 部分だけ loss をかける（`--mask-prompt`）。厳密な logit 蒸留ではなく、先生のお手本を SFT する形なので、実質は合成データ蒸留だ。

300 iter、数分で終わった。

```text
Trainable parameters: 0.171% (2.638M/1543.714M)
Iter  10: Train loss 4.469, Peak mem 2.519 GB
Iter 150: Val loss 2.592  ← ここが底
Iter 300: Train loss 0.475, Val loss 2.918
```
<!-- evidence: command="mlx_lm.lora --config configs/lora.yaml"; log="reports/train.log" -->

Val loss は 150 iter あたりが底で、その後じわっと上がる。89 件しかない。データが少ないと、こうなる。

![LoRA 学習ログと推論ベンチ](/images/slm-humor-distill-train-bench.png)

## 育った 1.5B

同じお題をもう一度聞いた。

当たりもあった。

> **満員電車で靴紐がほどけた** → 靴紐ほどけるなんて 満員電車の隙間探しより難しいねえ

> **エレベーターで同僚と二人きり** → 同僚に押されて奥に下がる「君の圧力測ったらどれくらいだろうね」

素の 1.5B からは、たしかに「外して笑わせる」方向に寄った。

一方で、盛大に滑った例も残しておく。ここが地味におもしろい。

> **会議でZoomのミュートを忘れていた** → 会議中はいつもミュートにしているのに今日の会議は…（延々と続く）

> **コーヒーをこぼした** → コーヒーのコーヒー除けになりましたね

繰り返しループと、意味の崩壊。1.5B の地力と、小さいデータの過学習が、そのまま出ている。

## で、CPU だけで動くのか

MLX は Metal（GPU）を使う。だから「MLX で動いた」は厳密には CPU-only ではない。

ただ、クラウド GPU なし・ローカル完結、という意味なら M5 + 32GB で十分だった。


| モデル              | 生成速度        | Peak memory |
| ---------------- | ----------- | ----------- |
| 素の 1.5B（4bit）    | 約 123 tok/s | 約 1.0 GB    |
| 蒸留後（fused, 4bit） | 約 122 tok/s | 約 1.0 GB    |

<!-- evidence: command="mlx_lm.generate --model fused_model/humor-1.5b-fused"; log="reports/bench_after.json" -->

育てても速度・メモリはほぼ変わらない。地味だけど、これは嬉しい。

### 本当に GPU を切ったら

厳密にやるなら GGUF + `llama.cpp` の `-ngl 0`。

```bash
llama-cli -m Qwen2.5-1.5B-Instruct-Q4_K_M.gguf -ngl 0 \
  -p "傘を忘れた。くすっと笑える一言をどうぞ。" -n 50
```


| 実行系                       | 生成速度        |
| ------------------------- | ----------- |
| MLX（Metal / GPU 使用）       | 約 122 tok/s |
| llama.cpp `-ngl 0`（純 CPU） | 約 64 tok/s  |

<!-- evidence: command="llama-cli -m ...Q4_K_M.gguf -ngl 0"; log="reports/cpu_only_run.log" -->

GPU を切っても 64 tok/s。半分くらい落ちるが、チャットとしては十分すぎる。

「CPU だけで会話できるか？」への答えは、1.5B クラスなら **Yes** だった。

## 限界

まず一番大きいのは、**ユーモアは定量評価できない**こと。tok/s は測れる。「面白さ」は測れない。

そのため、この記事で言えるのは次の範囲だ。

- 合成データ蒸留 + LoRA で、タスクの「型」には寄せられる
- 先生の質が天井になる（今回は 14B ローカル + 多言語スリップ）
- データ 104 件はデモ規模。過学習も出る
- LoRA で賢さの上限は上がらない。「寄せる」話ではある

一方で、まだ言えないこともある。

- 本番で使えるユーモア品質
- TAID のような logit 蒸留との優劣
- 汎用性能を落とさずに笑いだけ足す方法

次に進めるなら、教師データの件数と品質フィルタを積む。先生を 32B に上げるより、データの方が効きそうだ（ただし M5 で 32B 4bit は載る）。

## まとめ

振り返ると、こういう話だった。

- 「作る」を推論とタスク特化の蒸留に分けた
- 先生 14B → 生徒 1.5B を、M5 MacBook Air だけで回した
- 先生は中国語に滑った。フィルタで泥くさく掃除した
- 素の 1.5B は励ました。育てた後は、外して笑わせる方に寄った（滑ることもある）
- 蒸留しても推論速度はほぼ変わらなかった。CPU-only でも 64 tok/s

最初の「手元の Mac だけでモデル作れないか？」への答えは、こうなる。

**ゼロから賢いモデルを作る話ではない。でも、なけなしで買った Air の上で、1.5B を「この一言だけちょっと笑える相棒」に育てて、GPU なしのローカルで回す話なら、現実的な土俵だった。**

妻への弁明として十分かは、また上司とのレビューより厳しい別の審査がある。
